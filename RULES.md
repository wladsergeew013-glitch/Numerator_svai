# RULES — Нумератор свайного поля

## 1. Назначение проекта

Приложение предназначено для загрузки координат свай, разбиения точек на группы и нумерации свайного поля.

Рабочая архитектура:
- backend: Python + FastAPI;
- frontend: React + TypeScript + Vite + react-konva;
- рабочий формат проекта: `.pilenum.json`;
- CSV используется как формат импорта/экспорта, но не как основной формат хранения.

## 2. Главные сущности

### Точка / свая

У точки обязательно различать:
- `id` — внутренний постоянный идентификатор;
- `sourceId` — идентификатор из исходной выгрузки, если есть;
- `sourceNumber` — номер из исходного файла;
- `number` — текущий итоговый номер сваи;
- `groupId` — группа;
- `locked` — заморожена;
- `manualNumber` — номер задан пользователем;
- `syncState` — статус синхронизации.

`id` нельзя использовать как номер сваи.

### Группа

Группа хранит:
- `id`;
- `name`;
- `order`;
- `color`;
- `visible`;
- `locked`;
- `numbering`.

Порядок групп в диспетчере задаёт порядок сквозной нумерации.

## 3. Интерфейс CAD-типа

Обязательные элементы:
- верхняя ribbon-панель команд;
- рабочее поле;
- нижняя строка состояния;
- плавающие/закрепляемые окна;
- диспетчер групп;
- журнал изменений;
- информация о точке;
- настройки рабочего поля.

Верхняя панель должна:
- быть сверху;
- иметь группы команд;
- уметь растягиваться вниз;
- увеличивать кнопки пропорционально высоте;
- сохранять высоту между запусками.

## 4. Рабочее поле

Должны быть:
- zoom к курсору;
- pan;
- Zoom Extents;
- координатная сетка;
- малый и крупный шаг сетки;
- отдельные оси X/Y;
- цвет фона;
- базовые цвета фона;
- ручной выбор цвета фона;
- показ координат курсора;
- показ масштаба.

Настройки рабочего поля хранятся в проекте и дублируются в пользовательский localStorage-конфиг.

## 5. Отображение номеров

Должны быть настройки:
- показывать/скрывать номера точек;
- авто-контраст текста;
- ручной цвет текста;
- ручной цвет обводки;
- подсвечивать точки без номера.

Номера должны быть читаемыми на любом фоне:
- использовать контрастный текст;
- использовать обводку;
- использовать подложку под номер.

Точки без номера при включённой настройке подсвечиваются отдельным маркером.

## 6. Окна и docking

Плавающие окна:
- можно двигать за заголовок;
- можно закрывать;
- можно менять ширину боковыми ползунками;
- можно менять высоту нижним ползунком;
- можно менять ширину и высоту угловым ползунком;
- размер и позиция должны сохраняться в localStorage.

Диспетчер групп:
- может быть плавающим;
- может закрепляться слева;
- может закрепляться справа;
- при закреплении растягивается по высоте рабочего пространства;
- при закреплении сужает рабочее поле на свою ширину;
- при откреплении рабочее поле снова расширяется.

При подведении dock-окна к левому или правому краю должен появляться якорь закрепления.

## 7. Диспетчер групп

Диспетчер групп должен:
- создавать группу;
- удалять группу;
- выбирать активную группу кликом по карточке;
- менять имя;
- менять цвет;
- автоматически назначать новый свободный базовый цвет;
- сворачивать/разворачивать карточки;
- разворачивать все;
- сворачивать все;
- менять порядок групп;
- показывать количество точек в группе;
- задавать метод нумерации;
- задавать допуски;
- задавать направление.

Автоназначение включается/выключается командой **Назначить** на верхней ленте. Если при нажатии **Назначить** уже есть выделенные точки, они назначаются в активную группу один раз, без зажатия режима. Если выделения нет, команда переключает режим автоназначения рамкой.

Исключение: если открыто окно “Информация о точке”, автоназначение групп временно блокируется.

## 8. Журнал изменений

Журнал должен быть понятным пользователю, без сырого JSON.

Журнал показывает:
- импорт CSV;
- открытие проекта;
- создание группы;
- удаление группы;
- назначение точек;
- изменение порядка групп;
- нумерацию группы;
- нумерацию всех групп;
- сохранение проекта в папку projects.

## 9. Локальная папка проектов

Backend должен иметь папку `projects/`.

Кнопка “Проекты” должна:
- показать список проектов из папки `projects`;
- сохранить текущий проект в `projects`;
- открыть выбранный проект из списка.

Endpoints:
- `GET /api/projects/local`;
- `POST /api/projects/local/save`;
- `GET /api/projects/local/{file_name}`.

Если кнопка “Проекты” ничего не делает, сначала проверить, обновлён ли backend и запущен ли backend.

## 10. Нумерация

Должны быть две команды:
- нумеровать активную группу;
- нумеровать все группы.

Кнопки нумерации не должны выглядеть как постоянно зажатые/активные.

Рабочий метод MVP:
- `rows`;
- `columns`.

Будущие методы:
- `route`;
- `vector`;
- `manual`.

## 11. Безопасность UX

Нельзя терять уже сделанные UX-фичи при рефакторинге:
- CAD-like ribbon;
- draggable/resizable панели;
- docking диспетчера групп;
- настройки сетки;
- скрытие номеров;
- подсветка точек без номера;
- журнал изменений;
- окно информации;
- запрет автоназначения групп при активном окне информации;
- сохранение положения/размера окон;
- сохранение высоты toolbar;
- localStorage-конфиг пользователя;
- `.pilenum.json` как основной формат проекта.

## 12. Backend-структура

Backend должен оставаться компактным:
- `main.py`;
- `schemas.py`;
- `io_project.py`;
- `numbering_rows.py`;
- `numbering_route.py`;
- `numbering_vector.py`;
- `numbering_manual.py`;
- `clustering_auto.py`;
- `sync_import.py`.

Не плодить `routes/services/controllers/repositories`, пока это не станет реально нужно.

## 13. Правила выделения точек

Выделение рамкой работает в CAD-логике:
- протягивание рамки слева направо — зелёная область, точки добавляются к текущему выбранному набору;
- протягивание рамки справа налево — синяя область, точки удаляются из текущего выбранного набора;
- если включено автоназначение и активна группа, зелёная рамка назначает найденные точки в активную группу;
- синяя рамка только убирает точки из выбора и не меняет принадлежность к группе;
- в режиме окна “Информация о точке” автоназначение групп заблокировано.
- клавиша `Esc` очищает текущий выбранный набор точек, не меняя группы и номера;

## 14. Режим информации о точке

Когда открыто окно “Информация о точке”, рабочее поле переходит в режим выбора одной точки:
- клик по самой точке выбирает только её;
- клик по пустому месту выбирает ближайшую точку;
- при выборе точки её группа становится активной, если точка уже принадлежит группе;
- выделение рамкой и назначение точек в группу в этом режиме не выполняются.

## 15. Закрепляемые окна

Все основные окна должны работать одинаково:
- диспетчер групп;
- проекты;
- настройки рабочего поля;
- журнал изменений;
- информация о точке.

Окна должны:
- сохранять размер и положение после закрытия и повторного открытия;
- иметь крупные якоря закрепления слева/справа;
- при закреплении учитывать уже закреплённые окна и не занимать одно и то же место;
- при закреплении сужать рабочее пространство на свою ширину;
- при откреплении освобождать рабочее пространство.

## 13. Журнал операций как точка возврата

Журнал изменений — не только список событий. Для новых операций текущей сессии нужно хранить снимок состояния после операции.

Правила:
- клик по записи журнала возвращает точки, группы, настройки вида и выбранный набор к состоянию после этой операции;
- старые записи, загруженные из файла без снимка состояния, можно показывать как неактивные;
- очистка журнала очищает только список операций и снимки журнала, но не меняет точки, группы, номера и настройки;
- возврат через журнал не должен автоматически создавать новую запись, иначе журнал будет раздуваться.

## 14. Экспорт проекта

Кнопка сохранения проекта через браузер называется `Экспорт`, а не `Скачать`.

Смысл команды:
- сформировать текущий `.pilenum.json`;
- отдать его пользователю через браузер;
- не путать с сохранением в локальную папку `projects`, которое выполняется через окно `Проекты`.

## 15. Закреплённые окна и рабочее поле

Закреплённые окна должны сужать рабочее поле ровно на свою ширину.

Правила:
- между рабочим полем и закреплённым окном не должно быть лишнего пустого зазора;
- у закреплённых окон нельзя оставлять большую тень в сторону рабочего поля;
- плавающие окна могут иметь тень, закреплённые — нет;
- при закрытии окна оно должно убираться из dock-резерва;
- если окно неактивно/не отрисовано, оно не должно продолжать съедать рабочее поле.

## UI: active command highlighting

Команды верхней панели, которые работают как включаемые режимы/окна, должны иметь активное визуальное состояние, пока режим включён:

- `Группы` — активно, пока открыт диспетчер групп.
- `Инфо` — активно, пока открыт режим информации о точке.
- `Настройки` — активно, пока открыто окно настроек рабочего поля.
- `Сетка` — активно, пока включена сетка.
- `Номера` — активно, пока включён показ номеров точек.
- `Пустые` — активно, пока включена подсветка точек без номера.
- `Журнал` — активно, пока открыт журнал изменений.

Активная команда не должна выглядеть как случайно зажатая кнопка, но должна явно подсвечиваться рамкой/фоном/нижним индикатором.


## EXE / config discipline

Следующий этап — упаковка приложения в `.exe`, поэтому нельзя размазывать пользовательские настройки по случайным местам кода.

Правила:
- проектные данные хранятся в `.pilenum.json`;
- локальные проекты хранятся в папке `projects`;
- пользовательские UI-настройки должны быть вынесены в явный конфиг при переходе к exe-версии;
- сейчас браузерная версия временно хранит UI-настройки в `localStorage`, но это нужно рассматривать как адаптер хранения, а не как бизнес-логику;
- размеры/позиции окон, высота ribbon-панели, режимы выделения, цвет фона, сетка, показ номеров и подсветка пустых точек должны сохраняться через единый слой конфигурации;
- при упаковке в exe целевой файл конфига: `config/user_config.json`;
- дефолты допускается держать в коде только как fallback, но не как единственный источник пользовательской настройки.

## Dock registry / phantom spacing

Список активных закреплённых окон — runtime-состояние, а не постоянный пользовательский конфиг.

Правила:
- нельзя сохранять список активных dock-окон как вечный `localStorage`-резерв;
- при старте страницы dock-registry должен пересобираться только из реально отрисованных окон;
- если окно закрыто или не смонтировано, оно не должно съедать рабочее пространство;
- размеры и позиции окон сохраняются отдельно, но активный dock-резерв должен существовать только для открытых окон.

## File commands: open/import/save/export

Верхняя группа `Файл` должна различать четыре разные операции:

- `Открыть` — единая команда выбора файла. По расширению/содержимому определяет действие:
  - `.pilenum.json` / `.json` / JSON-объект — открыть проект;
  - `.csv` или не-JSON текст — импортировать исходные точки.
- `Сохранить` — сохранить текущий проект в локальную папку `projects` на backend. Это быстрый дубль команды из окна `Проекты`.
- `Экспорт` — скачать текущий проект как `.pilenum.json` через браузер.
- `Проекты` — открыть окно со списком локальных проектов из папки `projects`.

Нельзя снова разделять `Импорт` и `Открыть` на две верхние кнопки без отдельной причины: пользователь выбирает файл, приложение само понимает формат.

## 15. Правила имени проекта и локального сохранения

1. Единая команда `Открыть` сама определяет формат файла:
   - `.csv` — создать новый проект из исходной выгрузки;
   - `.json` / `.pilenum.json` — открыть готовый проект нумератора.
2. При открытии CSV имя нового проекта автоматически берётся из имени исходного файла без расширения.
3. CSV считается новым проектом: `project.fileName = null` до первого сохранения в папку `projects`.
4. При первом нажатии `Сохранить` для нового проекта приложение должно предложить имя проекта. После подтверждения проект сохраняется в локальную папку `projects`.
5. Имя файла в папке `projects` строится из текущего поля `Проект`.
6. Если проект уже сохранялся и пользователь изменил поле `Проект`, команда `Сохранить` сохраняет/перезаписывает файл с новым именем проекта.
7. Команда `Экспорт` не равна локальному сохранению: она только скачивает `.pilenum.json` через браузер.
8. Для будущей EXE-версии все такие настройки должны быть перенесены из `localStorage` в явные конфиги/файлы, чтобы поведение не зависело от браузерного профиля.

## 12. Desktop EXE and persistent user config

Desktop EXE must not rely only on browser/WebView localStorage for important UI state.

Persistent user settings are stored in a real file:

```text
config/user_config.json
```

For development/browser mode this folder is under the project root. For frozen EXE it must be next to `PileNumbering.exe`:

```text
dist/
  PileNumbering.exe
  projects/
  config/
    user_config.json
```

The config stores:

- background/grid/view settings;
- show/hide point numbers;
- highlight unnumbered points;
- selection rectangle mode;
- group manager visibility/dock/collapse state;
- collapsed group cards;
- toolbar height;
- floating/docked panel positions and sizes.

`localStorage` is allowed only as browser fallback/cache. The file config is the source of truth for EXE packaging.

Backend API:

```text
GET  /api/config/user
POST /api/config/user
```

When adding new UI settings, update both TypeScript defaults and backend `UserConfig` schema so EXE keeps the same behavior after restart.

## Дополнение v33 — ribbon и диспетчер групп

- В ribbon визуальная плитка иконки и подпись команды разделяются: рамка вокруг иконки, подпись отдельной строкой ниже.
- Нельзя делать подпись команды внутри той же рамки, где и иконка: на маленькой высоте ленты это становится нечитаемым.
- Команда **Назначить** живёт на верхней ленте, а не чекбоксом внутри диспетчера групп.
- Диспетчер групп должен быть читаемым рабочим окном: блок пайплайнов отдельно, блок действий отдельно, список групп отдельно.
- При широкой панели диспетчер обязан использовать ширину и раскладывать карточки/настройки в колонки.

### Дополнение v80 — dock viewport и индикатор осей

- Рабочее поле при закреплении окон считается как отдельный fixed-viewport между ribbon и строкой состояния.
- Закрепленные окна слева/справа сужают именно видимую область Canvas, а не накладываются поверх нее.
- Индикатор осей X/Y в левом нижнем углу выводится поверх Canvas как DOM-overlay внутри видимой области рабочего поля, поэтому он не должен пропадать под закрепленным окном.
- При закрытии/откреплении окна dock-reserve пересчитывается только по реально открытым dock-панелям.

### Дополнение v81 — dock reserve по фактическим DOM-панелям

- Рабочее поле не полагается только на временный dock-registry в localStorage.
- При открытии, закрытии, закреплении и изменении размера окон резерв рабочего поля пересчитывается по фактическим `.docked-panel.docked-left/.docked-right` в DOM.
- Если dock-registry ещё не успел обновиться, но панель уже видима, Canvas всё равно сдвигается за реальную границу панели.
- Дополнительный DOM-индикатор осей из v80 убран: используется один штатный Konva-индикатор X/Y внутри Canvas, без дублей.
